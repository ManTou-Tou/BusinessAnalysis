## 1. 产品背景

- **用户**：Shopee / TikTok Shop / Lazada 的电商卖家（运营人员）。
- **定位**：**内部运营后台**（admin 工具），非消费者前台；表格 / 表单 / 数据密集。
- **价值链**：卖家导入店铺数据 → AI Agent 自动分类评论、算利润、出每日报告 → 卖家审核并执行。
- **阶段**：MVP，后端已通「数据 + 三个 Agent（评论分类 / 利润 / 日报）+ 异步任务 + Excel 导入」。**无真实平台 API**，数据靠 Excel 导入或手工录入。

### 设计师必须理解的 4 个约束（直接影响交互）

1. **双层上下文常驻**：所有数据归属 **账号 → 店铺**。界面**始终显示/可切换「当前账号 + 当前店铺」**；几乎每个数据页都先依赖「选定店铺」。
2. **异步任务是第一交互模式**：评论分类、利润计算、日报、Excel 导入**都不是点完即出结果**——后端建任务、worker 跑、前端**轮询** `GET /agent/tasks/{id}` 看状态。必须设计「已提交 → 处理中 → 完成/失败」的反馈。**这是本产品最关键、最高频的交互**。
3. **占位鉴权**：用「账号 ID」标识身份，需要一个**轻量「账号切换/输入」入口**，不是完整登录。
4. **数据量大**：商品/订单/评论可成千上万 → keyset 分页、空态、加载态。

左侧导航 + 顶部全局栏的经典后台布局。

**顶部全局栏（常驻）：** 当前**账号**切换 ｜ 当前**店铺**下拉（切账号时重置）｜ 全局任务通知（完成/失败 toast）。

| **P0** | 商品管理 | 商品列表 + 详情（+ 触发利润、看利润） | products + profit ✓ |
| **P0** | 评论管理 | 评论列表 + AI 分类结果 + 触发分类 | reviews + classify ✓ |
| **P0** | 数据导入（Excel） | 上传 xlsx + 看导入任务结果 | imports ✓ |
| **P0** | 任务状态（详情/轮询） | 任一异步任务的进度与结果 | `GET /agent/tasks/{id}` ✓ |
| **P1** | 订单管理 | 订单列表 + 明细 | orders ✓ |

## 3. 全局 / 跨页面交互模式（请优先设计这几套「壳」）

### 3.1 账号 + 店铺上下文
- 账号：输入/选择账号 ID（占位鉴权），记住上次选择。
- 店铺：从该账号 `GET /shops` 列表选「当前店铺」；切店铺刷新依赖数据；切账号清空店铺。
- 空态：该账号还没店铺 → 引导去「数据导入（shops）」。

### 3.2 异步任务反馈（**本产品最重要的交互**）
适用：评论分类、利润计算、日报、Excel 导入。统一一套「任务结果卡 / 抽屉」，数据来自 `GET /agent/tasks/{id}` 轮询：
- **提交后**：提示「任务已提交」+ 任务 ID；触发按钮 loading/禁用，防重复提交。
- **处理中**：轮询状态 `pending / running`；导入额外有「已处理 N 行」进度（`output_json.progress`）。
- **完成 succeeded**：成功汇总（导入：新增 X / 更新 Y / 失败 Z）。
- **失败 failed**：`error_type` + 可读 `error_message`。
- **行级错误**：导入展示「坏行清单（行号 + 原因）」——坏行不阻塞好行入库；建议可查看/下载。
- **可取消**：仅 pending/retry 的任务可 `POST /agent/tasks/{id}/cancel`。
- 设计原则：**诚实**——不能让用户以为「点完就成了」，要表达「后台在跑，稍后看结果」。

> ⚠️ 当前**没有任务列表接口**：用户提交后要拿得到任务 ID（来自触发接口的返回），靠它查单个任务。跨任务的「历史列表」属 P1 且需后端新增（见 4.7）。

### 3.3 列表通用件
- 游标分页：「加载更多 / 下一页」，**无总页数/跳页**；`limit ≤ 100`。
- 空态、加载骨架、错误重试。
- **筛选要按真实参数收敛**（见 §0 对照表）：能服务端筛的只有 `shop_id` / `product_id`。其它维度（如评论按 type/情感）**当前只能对「已加载的当页数据」做前端筛选**，或标 [需新增后端] 需新增后端筛选——设计时请注明这一限制，别画成全量服务端筛选。

### 3.4 限流提示
- 创建 / 导入 / AI 触发类接口有 Redis 固定窗口限流（按 路由 + 账号；无账号按 IP；Redis 异常时放行）。超限返回「请稍后再试（含等待秒数）」→ 需友好「操作太频繁」提示，而非报错弹窗。（GET 读取类一般不限流。）

## 4. 各页面 feature 清单

> 每页给：**目的 / 关键数据字段 / 主要操作 / 必需状态**。字段来自后端实际模型；[需新增后端] = 后端暂无、需新增。

### 4.1 概览 / 每日报告〔P0〕
- **目的**：看**当前店铺某一天**的表现 + AI 总结。**注意：是「单店 + 单日」**，不是跨店总览。
- **数据**（`GET /analytics/shop/{id}/daily` + `/report`，按 shop + date）：
  - 数字指标：订单数、营收、利润、广告花费、销量；利润**覆盖度**（多少明细有成本算进利润，缺成本的不当 0）。
  - 曝光 / 点击 / 转化率：**暂无数据源 → 显示「暂无」**，禁止编造。
  - AI「**当日总结**」（一段话）+「**建议行动**」（要点）+ 确定性 `sections_json`（若含 Top 商品/低利润商品等就展示）。
  - **Top 商品 / 趋势 / 跨店对比**：[需新增后端] 无专门接口——**仅当日报 `sections_json` 里有才显示**，否则留位标「需新增」。
- **操作**：选日期（默认昨天）；「生成某日报告」（触发异步 `POST /agent/daily-report` → 进 §3.2 轮询）；查看历史某日。
- **状态**：报告未生成（引导生成）/ 生成中 / 已生成。**数字区**（权威）与 **AI 散文区**（辅助解读）视觉上要分权重，避免用户把 AI 措辞当精确数字。

### 4.2 商品管理〔P0〕
- **目的**：查看/维护商品，看每个商品真实利润。
- **列表字段**（`GET /products?shop_id=`）：商品名、SKU、类目、售价、成本、库存、状态(active/inactive/out_of_stock)。**列表仅能按 shop_id 拉**，无服务端状态/搜索筛选（见 §3.3）。
- **详情字段**（`GET /products/{id}`）：上述全部；利润聚合来自 `GET /analytics/product/{id}/profit`（毛利/净利/利润率/盈亏平衡价；**缺成本的部分标注「未计入」**，不当 0）。
- **操作**：新增（`POST`）、编辑（`PUT`，**无删除**）；按店铺看；详情里「触发利润计算」（异步 `POST /agent/profit-analysis` → §3.2）后再看利润。
- **状态**：分页加载、空态（引导导入）、库存 0 / 下架标识、利润待计算 / 计算中 / 已算。
- ⚠️ **不要画删除、批量删除、按状态服务端筛选**（后端不支持）。

### 4.3 评论管理〔P0〕
- **目的**：看评论 + AI 分类结果，决定关注质量/物流问题。
- **列表字段**（`GET /reviews?shop_id=&product_id=`）：买家名、评分(1–5)、内容、**分类(review_type)**、**情感(sentiment)**、所属商品、时间。
  - review_type：positive / negative / logistics_issue / quality_issue / price_issue / feature_question / spam。
  - sentiment：positive / neutral / negative。
  - **筛选**：服务端**只有 shop_id / product_id**；按 type/情感/评分筛 → 前端当页筛或标 [需新增后端]。
- **操作**：
  - **触发 AI 分类（均为异步，进 §3.2 轮询，不即时出结果）**：单条 `POST /agent/review-classifier`；批量 `POST /reviews/classify`（分类本店未分类评论）。
  - 区分「未分类 / 已分类」。
- **状态**：未分类（灰）/ 分类中 / 已分类（用彩色标签区分 type 与 sentiment）；负面 / 投诉类醒目，便于优先处理。
- **注意**：投诉类不自动回复（回复功能属未来阶段，本期不画）；**无单条详情页/编辑/删除接口**。

### 4.4 数据导入（Excel）〔P0〕
- **目的**：把 products / orders / reviews / shops 用 `.xlsx` 批量导入。
- **feature**：
  - 选导入实体（商品/订单/评论/店铺）；**非「店铺」必须先选目标店铺**（带 `shop_id`）。
  - 上传 `.xlsx`（拖拽 + 点选）；**仅接受 .xlsx**、限大小（默认 20MB）；展示各实体模板列要求（必填列不同，见 backend/docs/import_templates.md）。
  - **冲突策略提示**：products / orders / shops = 「更新或新增(upsert)」；reviews = 「仅追加(insert / append-only)」（reviews 传 upsert 会被拒）。
  - 上传后返回**异步任务**（`AgentTaskOut`）→ §3.2：进度「已处理 N 行」、结果「新增/更新/失败」、**坏行清单**。
  - **orders 也在本页导入**（不要求先做订单管理页）。
- **状态**：上传中 / 排队 / 解析中 / 完成 / 失败；失败可重试（评论导入**断点续跑**，重试不重复；商品/订单/店铺从头重跑但幂等）。

### 4.5 任务状态（详情 / 轮询）〔P0〕
- **目的**：任一异步任务（导入/分类/利润/日报）提交后，凭任务 ID 看进度与结果。**这是 P0 闭环的必需件。**
- **数据**（`GET /agent/tasks/{id}`）：任务类型、状态(pending/running/succeeded/failed/retry/cancelled)、店铺、创建/完成时间、`output_json`（结果摘要 / 行级错误）、`error_type/message`、重试次数。
- **操作**：轮询刷新；可取消（仅 pending/retry）。
- 形式可为「全局任务抽屉」或「触发处内联结果卡」（见 §7 开放问题）。

### 4.6 订单管理〔P1〕
- **目的**：查看订单及明细。
- **列表字段**（`GET /orders?shop_id=`）：订单号、状态(pending/paid/shipped/completed/cancelled/refunded)、总额、时间。**仅按 shop_id 拉**，无状态/日期服务端筛选。
- **详情**（`GET /orders/{id}`）：订单头（运费/平台费/广告费）+ **多条明细**（商品/数量/单价/该明细利润）。
- **操作**：按店铺看、查看明细。**无 update/delete**。

### 4.7 任务中心（历史列表）〔P1，[需新增后端] 需新增后端〕
- **目的**：跨任务的状态列表 + 按 类型/状态/店铺 筛选，便于排障。
- **现状**：后端**只有按 ID 查单个任务**，**没有列表接口**。要做此页需先新增「按 account/shop/status/task_type 的分页查询接口」。**P0 先用 4.5 的单任务轮询闭环；此页留 P1。**

### 4.8 店铺管理〔P2〕
- 列表（平台、店铺名、外部店铺 ID、状态）+ 新增/查看（`POST/GET`）。**无 update/delete**（改店铺信息当前只能靠 Excel 导入 upsert）。
